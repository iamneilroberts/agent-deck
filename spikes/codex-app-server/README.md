# Spike A ‚Äî Codex app-server

**Phase 0 spike.** Drives the real `codex app-server` over stdio end to end to settle *what
the protocol actually is* before AgentDeck's UI and adapter are built around assumptions.

Verified against **`codex-cli 0.144.1`**, Node 22.12, Linux, ChatGPT login, on 2026-07-13.

## Run it

```bash
cd spikes/codex-app-server
npm install
npm run test        # pure-unit: framing + redaction (no agent needed)
npm run typecheck
npm run spike        # drives the REAL codex app-server (needs `codex` on PATH + logged in)
```

`npm run spike` performs a full lifecycle and writes two transcripts to `captured/`:
`session-<ts>.md` (sanitized, committed) and `raw-<ts>.jsonl` (full wire log, **gitignored**).
Nothing binds to a socket ‚Äî the spike talks to app-server over stdio only.

## What the spike demonstrates (all against the real binary)

1. `initialize` + `initialized` handshake.
2. `thread/start` ‚Üí thread id.
3. `turn/start` with live streaming (`agentMessage` deltas, item lifecycle, `commandExecution`).
4. A real **approval round-trip**: server sends `item/commandExecution/requestApproval`; the
   spike replies with a decision drawn from the server's own `availableDecisions`.
5. `turn/interrupt` on a second, longer turn (settles `status: "interrupted"`).
6. `thread/list` + `thread/resume` (the recovery primitives).
7. Clean shutdown.

Latest run: `captured/session-2026-07-13T07-36-19-861Z.md`.

## What app-server supports (findings)

- **Transport:** NDJSON over stdio (`--listen stdio://` default; also `unix://`, `ws://`).
  One JSON object per `\n`; multiple objects can share a chunk ‚Üí line-buffered parser
  (`src/framing.ts`).
- **Envelope ‚Äî NOT JSON-RPC 2.0.** Codex's own shape:
  - client‚Üíserver request `{ id, method, params }`
  - response `{ id, result }` / `{ id, error }`
  - notification (both directions) `{ method, params }`
  - server‚Üíclient request (approvals) `{ id, method, params }`
- **Two independent ID spaces.** The server initiates its own requests starting at `id: 0`
  while the client uses its own counter. Route inbound by presence of `method`, not by id;
  reply to a server request with *its* id. (`classify()` in `src/framing.ts`.)
- **v2 `thread`/`turn` API.** `thread/start` returns the thread at `result.thread.id` plus
  resolved `model`, `runtimeWorkspaceRoots`, `instructionSources`, `approvalPolicy`, `sandbox`.
  The app-server boots the user's `~/.codex` MCP servers per thread (visible as
  `mcpServer/startupStatus/updated` chatter ‚Äî suppressed in the spike log).
- **`turn/start` resolves on ACCEPT, not completion.** The response comes back as soon as the
  turn is accepted; the turn's real end is a `turn/completed` notification (`status`:
  `completed` | `interrupted` | ‚Ä¶). *Awaiting the request response is a bug* ‚Äî the adapter
  must await the notification, keyed by the `turnId` from `turn/started`. (This spike caught
  exactly that mistake in its first draft.)
- **Approvals are faithful and self-describing.** The request carries `command`, `cwd`,
  `commandActions`, an optional `proposedExecpolicyAmendment`, and **`availableDecisions`** ‚Äî
  the exact valid decisions for this prompt. The UI renders only those; it never invents a
  scope. Decision values: `"accept"`, `"acceptForSession"`, `"decline"`, `"cancel"`, or
  `{ acceptWithExecpolicyAmendment: {‚Ä¶} }`. A `serverRequest/resolved` notification confirms.
- **execpolicy auto-approves safe commands.** Under `read-only` + `on-request`, a trivially
  safe `echo` runs with **no** approval request; approval fires only when the action is
  actually blocked (e.g. a write under `read-only`) or under `untrusted`. The adapter must not
  assume every command yields an approval.
- **Item types map cleanly to timeline cards** (`ThreadItem`): `userMessage`, `agentMessage`
  (`phase: commentary | final_answer`), `reasoning`, `commandExecution` (command/cwd/
  processId/status/aggregatedOutput/exitCode/durationMs), `fileChange`, `mcpToolCall`,
  `webSearch`, `plan`.
- **Recovery works.** Threads are UUIDv7 persisted under `~/.codex/sessions/‚Ä¶jsonl`; a fresh
  process can `thread/list` (filterable by `cwd`) and `thread/resume` by id.
- **Protocol types are generated by the binary itself** ‚Äî
  `codex app-server generate-ts --out <dir>` and `‚Ä¶ generate-json-schema --out <dir>`. No
  reverse engineering; a version bump is regenerate-and-reconcile. `src/proto.ts` vendors only
  the hand-curated subset this spike uses.

## What remains uncertain (for later phases)

- **`thread/list` timing/consistency.** In one buggy run (turns still in flight) our just-
  created thread was briefly absent from `thread/list`; once turns settled it appeared. Needs
  a deliberate test of list-vs-active-thread consistency and the `useStateDbOnly` flag before
  Phase 2 relies on it for recovery UI.
- **Reconnect to a *running* thread from a second process.** `thread/resume` docs describe
  rejoining an active thread and treating a path as a consistency check. Not yet exercised
  across two concurrent app-server clients ‚Äî Phase 2 recovery needs this proven.
- **Full approval matrix.** Only `item/commandExecution/requestApproval` was exercised live.
  `item/fileChange/requestApproval`, `item/tool/requestUserInput`,
  `item/permissions/requestApproval`, and `mcpServer/elicitation/request` still need capture.
- **Managed process model.** One app-server per thread vs. one shared process multiplexing
  threads (the protocol supports many threads per connection) ‚Äî pick in Phase 2. The per-
  thread `~/.codex` MCP boot cost argues for measuring startup latency.
- **Cancellation/interrupt races.** Interrupt worked, but the exact ordering guarantees
  between `turn/interrupt`, in-flight approval requests, and `turn/completed` need a dedicated
  test.
- **Auth/token surface.** `account/*` notifications and `attestation/generate` /
  `account/chatgptAuthTokens/refresh` server requests exist; the redactor already masks their
  payloads, but the refresh flow is unhandled here and must be handled (or explicitly declined)
  in the adapter.

## Recommended changes to the later plan

1. **Codex adapter: model a turn as fire-then-await-notification**, never await the
   `turn/start` response for completion. Bake `turnId` correlation into the adapter's turn
   state machine.
2. **Drive the approval UI from `availableDecisions`** verbatim ‚Äî do not hardcode a decision
   set. This is the concrete mechanism behind ADR-0001's "never a silent approver."
3. **Adopt generated protocol types** as a build step (`generate-ts`) rather than the vendored
   subset, once the adapter package exists ‚Äî pin them per Codex version in CI.
4. **Add Spike A‚Ä≤ (small):** two concurrent clients resuming one running thread, plus the full
   approval matrix capture. Cheap, and it de-risks Phase 2 recovery + Phase 5 handoff (where a
   second agent process is the whole point).
5. **Consider disabling per-thread `~/.codex` MCP autostart** for AgentDeck-managed threads
   (config override) to cut cold-start latency, unless those servers are wanted.
