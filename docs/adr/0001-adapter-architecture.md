# ADR-0001: Vendor-neutral adapter boundary; Codex via app-server, Claude via stream-json

- Status: Accepted
- Date: 2026-07-13
- Context verified against: `codex-cli 0.144.1`, Node 22.12

## Context

AgentDeck controls two coding agents with different control surfaces. The server and UI must
not be coupled to either vendor's protocol, or every UI change risks churning against vendor
internals and a second agent becomes expensive to add. The brief also assumed Codex and Claude
have asymmetric capabilities (Codex structured, Claude terminal-only). Empirical checks change
that picture.

## Decision

1. **One vendor-neutral core.** The server and frontend depend only on `AgentAdapter`, a
   normalized `AgentEvent` union, and neutral session/approval types. Vendor-specific types
   live *inside* adapter packages and never leak upward. (Interface in
   [`architecture.md` §4](../architecture.md).)

2. **Codex adapter → `codex app-server` over stdio.** Verified: NDJSON framing, Codex's own
   `{id,method,params}` / `{id,result}` envelope (not JSON-RPC 2.0), the v2 `thread`/`turn`
   API, server-initiated approval requests that carry `availableDecisions`, and machine-
   generated protocol types (`codex app-server generate-ts`). We do **not** scrape the TUI.

3. **Claude adapter → headless Agent SDK as primary; PTY as fallback.** The brief's PTY-first
   design is downgraded and now **verified by Spike B** (`claude` v2.1.207, SDK 0.3.207): the
   SDK's `query()` gives structured streaming (`system/init` → `assistant`/`tool_use` →
   `tool_result` → `result`), `options.resume` recovers a session (context survived live), and
   **programmatic approval is the `canUseTool` callback** (`{behavior:"allow"|"deny",
   updatedInput}`), which wraps the CLI's control-protocol `can_use_tool` message. `node-pty`
   is retained only for genuinely interactive TUI moments. **Correction:** the
   `--permission-prompt-tool` flag the brief assumed is **removed** in v2.1.207 — the SDK
   `canUseTool` callback is the mechanism. AgentDeck-managed sessions set `settingSources: []`
   (or a dedicated `CLAUDE_CONFIG_DIR`) so the operator's global hooks/CLAUDE.md don't bleed in.

4. **Faithful transport, never re-implementation, never a silent approver.** Adapters map the
   agent's own events/approvals into neutral form and pass approvals through verbatim — the UI
   presents only decisions the agent offers (Codex hands us `availableDecisions`). Any
   AgentDeck-side convenience approval is explicit, scoped, logged, revocable, off by default.

5. **Two ID spaces on the Codex wire.** Client request IDs and server→client request IDs are
   independent (server observed starting at `id:0`). Transport routes inbound messages by
   presence of a `method` field and replies to a server request with its own id.

## Consequences

- Adding a third agent = one new adapter package; the core and UI are untouched.
- Codex integration risk is low: protocol types are generated by the binary, so a Codex
  version bump is a regenerate-and-reconcile, not a reverse-engineering exercise.
- Claude Phase 4 is materially de-risked and its adapter can mirror the Codex adapter's shape
  instead of being a fragile parser — **confirmed by Spike B** (`spikes/claude-headless/`).
- The neutral event model, not raw terminal bytes, is canonical. Unrecognized output is
  preserved as raw rather than guessed, so unknown states degrade to the terminal escape
  hatch instead of producing wrong structured cards.

## Alternatives considered

- **Scrape the Codex/Claude TUIs.** Rejected: fragile, and unnecessary given app-server and
  stream-json.
- **Expose each vendor protocol directly to the UI.** Rejected: couples the UI to vendor
  internals and makes a second agent expensive.
- **PTY-first for Claude (per the original brief).** Rejected as the primary path; kept as
  fallback. Structured stream-json is more robust for the conversational surface.
